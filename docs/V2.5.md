# ZeroViza V2.5 — Fully Decentralized Architecture

> SQLite eliminated. All data now lives on 0G Storage + smart contracts.
> Zero filesystem dependency. Vercel-ready. Truly on-chain.

---

## What Changed

V2.5 replaces the centralized SQLite database with two smart contracts and 0G Storage. Every piece of user data — chat history, profiles, documents, lawyer applications — is now stored on decentralized infrastructure.

| Component | V1 (SQLite) | V2.5 (0G + Contracts) |
|-----------|------------|----------------------|
| Storage index | `storage_index` table in SQLite | `StorageIndex.sol` contract on 0G Galileo |
| Document metadata | `documents` table in SQLite | JSON array on 0G Storage, root hash in `StorageIndex.sol` |
| Lawyer applications | `lawyers` table in SQLite | Profile JSON on 0G Storage, status in `LawyerRegistry.sol` |
| Lawyer verification | SQLite status column + optional contract | `LawyerRegistry.sol` is the single source of truth |
| Database dependency | `better-sqlite3` (native C++ module) | None — removed entirely |
| Vercel compatibility | Partial (`/tmp` ephemeral, data lost between deploys) | Full — no filesystem writes |

## New Smart Contracts

### StorageIndex.sol

**Address**: `0xbBb868BcA991c8C9e184F236bD7AfAB79e4F602b` (0G Galileo)

Maps each wallet to three 0G Storage root hashes:

```
wallet --> { historyRoot, profileRoot, documentsRoot }
```

- `historyRoot` — JSONL of chat messages on 0G Storage
- `profileRoot` — JSON user profile (streak, preferences) on 0G Storage
- `documentsRoot` — JSON array of document metadata on 0G Storage

**Key design decisions:**
- **Operator pattern**: The server wallet is registered as an `operator` and can update mappings on behalf of any user. Users never need to sign transactions.
- **Partial updates**: Passing an empty string for any root leaves it unchanged — only the modified root is overwritten.
- **Gas efficiency**: One `updateRoots()` call updates all three roots in a single transaction.

```solidity
function updateRoots(
    address user,
    string calldata historyRoot,
    string calldata profileRoot,
    string calldata documentsRoot
) external onlyAuthorized;

function getRoots(address user) external view returns (
    string memory historyRoot,
    string memory profileRoot,
    string memory documentsRoot
);
```

### LawyerRegistry.sol (V2)

**Address**: `0x009158249E904A7089f8649ABb9b9268780E2D9a` (0G Galileo)

On-chain verification registry for accredited immigration lawyers. V2 adds:

| Addition | Purpose |
|----------|---------|
| `operator` state variable | Server wallet can submit applications on behalf of users |
| `applyOnBehalf(wallet, metadataURI)` | Operator submits application for a wallet |
| `getRecord(wallet)` | Returns full record: status, metadataURI, appliedAt, verifiedAt, rejectionReason |
| `_applicants[]` array | Tracks all applicants (not just verified) for admin enumeration |
| `applicantCount()` / `applicantAt(i)` | Iterate over all applications |

**Lawyer application flow (V2.5):**

```
1. User submits form on /lawyers/apply
2. Server uploads profile JSON to 0G Storage --> rootHash
3. Server calls LawyerRegistry.applyOnBehalf(wallet, rootHash)
4. Admin calls /api/lawyers/verify with { wallet, action: "approve" }
5. Server calls LawyerRegistry.verifyLawyer(wallet)
6. Lawyer appears in /lawyers directory (data read from chain + 0G)
```

**Verification status lives on-chain permanently:**
```solidity
enum Status { None, Pending, Verified, Rejected }

function getRecord(address wallet) external view returns (
    Status   status,
    string memory metadataURI,
    uint256  appliedAt,
    uint256  verifiedAt,
    string memory rejectionReason
);
```

## Data Flow Architecture

```
                    User (Wallet)
                         |
                    Next.js API Routes
                    /api/chat, /api/upload, /api/lawyers/...
                         |
              +----------+----------+
              |                     |
     Smart Contracts           0G Storage
     (mutable pointers)        (immutable data)
              |                     |
   StorageIndex.sol          Chat History (JSONL)
   wallet -> rootHashes      User Profiles (JSON)
                             Document Lists (JSON)
   LawyerRegistry.sol       Lawyer Metadata (JSON)
   wallet -> status +        Uploaded Files (binary)
             metadataURI
```

**Read path** (e.g., loading documents):
1. Call `StorageIndex.getRoots(wallet)` — 1 RPC call
2. Download documents JSON from 0G using `documentsRoot` hash
3. Parse and return

**Write path** (e.g., uploading a document):
1. Upload file to 0G Storage — get file `rootHash`
2. Download current documents list from 0G
3. Append new document metadata
4. Upload updated list to 0G — get new `documentsRoot`
5. Call `StorageIndex.updateRoots(wallet, "", "", documentsRoot)` — 1 tx

## Updated API Routes

| Method | Route | V2.5 Changes |
|--------|-------|--------------|
| POST | `/api/chat` | `await getStorageIndex()` / `await upsertStorageIndex()` (was sync SQLite) |
| GET | `/api/history` | `await getStorageIndex()` (was sync) |
| GET/PATCH | `/api/profile` | `await getStorageIndex()` / `await upsertStorageIndex()` (was sync) |
| POST | `/api/upload` | `await insertDocument()` — uploads doc list JSON to 0G + updates contract |
| GET/DELETE | `/api/documents` | `await getDocuments()` / `await deleteDocument()` — reads from 0G via contract |
| GET | `/api/lawyers` | `await getVerifiedLawyers()` — iterates contract + downloads metadata from 0G |
| POST | `/api/lawyers/apply` | Uploads metadata to 0G, calls `applyOnBehalf()` on contract |
| POST | `/api/lawyers/verify` | Now takes `{ wallet, action }` instead of `{ id, action }`. Writes directly to contract. |
| GET | `/api/lawyers/status` | `await getLawyerByWallet()` — reads from contract + 0G |

**Breaking change**: `/api/lawyers/verify` now uses `wallet` (not `id`) as the lawyer identifier. The wallet address is the canonical ID in V2.5.

## Contract Deployment

### Prerequisites

- [Foundry](https://book.getfoundry.sh/getting-started/installation) installed
- Server wallet funded with 0G testnet tokens from [faucet.0g.ai](https://faucet.0g.ai)

### Deploy

```bash
cd contracts

# Run tests first
forge test -vvv

# Deploy both contracts
forge script script/Deploy.s.sol \
  --rpc-url og_galileo \
  --broadcast \
  --legacy \
  --gas-price 3000000000 \
  --private-key $OG_SERVER_PRIVATE_KEY
```

The deploy script outputs both addresses. Add them to `.env.local`:

```env
NEXT_PUBLIC_STORAGE_INDEX_ADDRESS=0x...
NEXT_PUBLIC_LAWYER_REGISTRY_ADDRESS=0x...
```

## Trade-offs and Considerations

| Concern | Status | Notes |
|---------|--------|-------|
| Read latency | Slightly higher | Contract RPC call (~200ms) + 0G download (~500ms) vs SQLite (~1ms). Acceptable for demo/testnet. |
| Write latency | Higher | Each write requires an on-chain tx (~3-5s confirmation). Chat history updates are the most affected. |
| Gas costs | Minimal on testnet | Each `updateRoots()` call costs ~0.00000003 0G. Production mainnet costs TBD. |
| Concurrent writes | Last-write-wins | Two simultaneous requests for the same wallet may conflict. Acceptable for single-user-per-wallet model. |
| Enumeration cost | Linear | Listing all lawyers requires N RPC calls. Fine for <100 lawyers. Batch with multicall for scale. |
| Data migration | Clean start | V2.5 contracts are fresh — no migration from old SQLite data needed on testnet. |

## What This Enables

1. **Full Vercel compatibility** — no filesystem dependency, no ephemeral `/tmp` issues
2. **True decentralization** — every piece of data is on 0G Storage or on-chain
3. **Verifiable lawyer registry** — anyone can read `LawyerRegistry.getRecord(wallet)` directly from the chain
4. **Data portability** — users own their data via wallet address, not locked in a server database
5. **Censorship resistance** — no single server can delete user data or lawyer credentials

## Stack (V2.5)

- **Framework**: Next.js 15.5 (App Router), React 19
- **CSS**: Tailwind v4 with `@theme inline`
- **Auth**: Wagmi v2 + RainbowKit v2 — 0G Galileo (Chain ID 16602)
- **AI**: `qwen/qwen-2.5-7b-instruct` via 0G Compute Network
- **Storage**: `@0glabs/0g-ts-sdk` — content-addressed decentralized file storage
- **Data Layer**: `StorageIndex.sol` + `LawyerRegistry.sol` (Foundry/Solidity 0.8.24)
- **Contract Integration**: ethers.js 6.16 (server-only)
- **State**: Zustand + TanStack Query
- **Animation**: Framer Motion

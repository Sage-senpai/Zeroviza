# ZeroViza — System Architecture

> How a consumer AI immigration advisor uses **three 0G pillars end-to-end** with zero centralized infrastructure.

---

## One-sentence summary

ZeroViza is an AI-driven immigration legal aid dApp where every byte of user data lives on **0G Storage**, every AI response is generated by **0G Compute**, and every mutable pointer is secured by **0G Chain smart contracts**.

## Why this architecture?

Immigration data is **high-sensitivity**: passport numbers, asylum status, family situations, financial records. A centralized service is the wrong home for this data because:

1. **It can be subpoenaed** by governments users are fleeing from
2. **It can be breached** (every big breach ends up on dark markets)
3. **It can be shut down** (think of what happens to activists when the server disappears)
4. **It trains models silently** (your questions become someone else's training data)

0G solves all four:

- **0G Storage** — content-addressed, tamper-proof, no central host to subpoena
- **0G Compute** — inference provers can run in TEEs (Sealed Inference) so prompts never leak
- **0G Chain** — immutable pointer state; apps can shut down but user data persists

Users get the **same UX as ChatGPT** with the **privacy guarantees of a pencil and paper**.

## The operator pattern (why users don't pay gas)

Most dApps make users sign every transaction. For an immigration app serving people who may have never used crypto before, that's a non-starter.

ZeroViza uses an **operator pattern**:

1. A dedicated **server wallet** is registered as `operator` on both smart contracts
2. The server wallet pays gas for all user operations (contract writes + 0G Storage + 0G Compute)
3. Users only need a wallet for **identity** — no tokens required, no gas fees

```solidity
// StorageIndex.sol
function updateRoots(
    address user,
    string calldata historyRoot,
    string calldata profileRoot,
    string calldata documentsRoot
) external onlyOperator { ... }
```

This is the unlock that lets us onboard immigrants and asylum seekers who could never afford gas fees.

## Data model

### StorageIndex.sol

Maps each user wallet to **three 0G Storage root hashes**:

```solidity
struct Roots {
    string historyRoot;      // JSONL chat messages
    string profileRoot;      // JSON user profile + streak
    string documentsRoot;    // JSON array of document metadata
}
mapping(address => Roots) private _roots;
```

- **Partial updates** supported (empty string = leave unchanged) for gas efficiency
- **Content-addressed** — the root hash IS the retrieval key for 0G Storage
- **Mutable pointer** — as users add chat messages, the root hash updates on-chain

### LawyerRegistry.sol

On-chain lawyer verification with enumeration:

```solidity
enum Status { None, Pending, Verified, Rejected }
struct LawyerRecord {
    Status  status;
    string  metadataURI;      // 0G Storage root hash of profile JSON
    uint256 appliedAt;
    uint256 verifiedAt;
    string  rejectionReason;
}
```

- Lawyers apply via `applyOnBehalf()` (operator-signed, zero gas for lawyer)
- Admin approves/rejects via `verifyLawyer()` / `rejectLawyer()`
- Enumerable via `applicantCount()` + `applicantAt(i)`
- Profile metadata (name, specializations, jurisdiction, languages, bio) lives on 0G Storage — cheap to update, never touches a database

## Data flow: "user asks a question"

```
1. User types: "Can I apply for asylum in Germany?"
   │
   ▼
2. POST /api/chat  { message, walletAddress }
   │
   ▼
3. Server: read StorageIndex.getRoots(user).historyRoot
   │
   ▼
4. Server: download JSONL history from 0G Storage → last 10 messages
   │
   ▼
5. Server: call 0G Compute broker
      - getRequestHeaders(provider, body)  ← signed by operator wallet
      - getServiceMetadata(provider)       ← discover endpoint
      - POST {endpoint}/v1/proxy/chat/completions
   │
   ▼
6. AI response returned to user IMMEDIATELY (no wait for storage)
   │
   ▼
7. Background: append user+AI messages to history JSONL
   │
   ▼
8. Background: uploadHistory(messages) → new rootHash on 0G Storage
   │
   ▼
9. Background: StorageIndex.updateRoots(user, newHistoryRoot, "", "")
   │
   ▼
10. Background: broker.inference.processResponse() → fee settlement on-chain
```

**Key insight**: the user gets their AI response in ~2 seconds. The slow steps (0G Storage upload + on-chain write) run in the background via fire-and-forget. The next chat load pulls the updated history.
